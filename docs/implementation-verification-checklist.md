# План проверки соответствия реализации

Этот документ превращает `implementation_plan.md` в подробный чеклист приемки. Его задача - помочь проверить, что проект действительно реализован по плану, а не только формально закрыт по задачам.

## Как пользоваться документом

- [ ] Для каждого пункта зафиксирован статус: `passed`, `failed`, `blocked` или `n/a`.
- [ ] Для каждого пункта приложены доказательства: ссылка на код, тест, документ, лог, скриншот или артефакт run.
- [ ] Для каждого `failed` и `blocked` пункта записан конкретный дефект, риск или причина блокировки.
- [ ] Фаза считается пройденной только после прохождения всех ее пунктов и ее секции `Проверка фазы`.
- [ ] Если пункт проверяется вручную, рядом сохранен краткий сценарий воспроизведения и ожидаемый результат.

## Сквозные критерии приемки для каждой фазы

- [ ] Все декомпозированные задачи фазы реально реализованы в коде, конфигурации и пользовательских сценариях.
- [ ] Для нового и измененного кода есть автоматические тесты нужного уровня: unit, integration, regression или e2e.
- [ ] Пройден ручной smoke или e2e-сценарий по основному пользовательскому пути фазы.
- [ ] Обновлены документация, конфигурация, миграции данных, manifest-схемы, UX-тексты и сообщения об ошибках.
- [ ] Не осталось legacy-зависимостей, блокирующих следующую фазу.
- [ ] Все выводы проверки сохранены в виде артефактов: отчеты тестов, примеры run, fixtures, screenshots, manifests.

## Фаза 3. Переработка анализа сценария: от `3 keywords` к `paragraph intent`

### 3.1 Paragraph-first как ядро ingestion

- [ ] Paragraph-first модель сохранена как ядро системы и отражена в коде, данных и UI.
- [ ] Логика чтения нумерованных абзацев перенесена из `keyword_extractor.py` в shared ingestion service.

### 3.2 Поля, сохраняемые по абзацу

- [ ] Сохраняется и отображается `header_text`.
- [ ] Сохраняется и отображается `paragraph_no`.
- [ ] Сохраняется и отображается `original_index`.
- [ ] Сохраняется и отображается исходный текст абзаца.

### 3.8 Новый контракт данных

## Фаза 4. Storyblocks session automation core

### 4.1 Базовый automation stack

- [ ] Для Storyblocks используется `Playwright for Python` через `launch_persistent_context(...)`.
- [ ] Для каждого профиля используется dedicated `user_data_dir`.
- [ ] Используется реальный установленный browser channel: `chrome` по умолчанию, `msedge` как fallback.
- [ ] В документации явно записано, что временный Chromium/guest context, `codegen` и `storage_state` не являются поддерживаемым login baseline.

### 4.2 Manager браузерных профилей

- [ ] Можно создать профиль.
- [ ] Для профиля создается предсказуемая структура каталогов: `user_data`, `downloads`, `diagnostics`.
- [ ] Можно выбрать профиль.
- [ ] Можно переименовать профиль.
- [ ] Можно удалить профиль.
- [ ] Можно переключить активный профиль.

### 4.3 Session manager

- [ ] Браузер открывается через persistent context.
- [ ] Перед запуском проверяется доступность выбранного browser channel.
- [ ] Детектируется ситуация, когда профиль занят другим браузерным процессом.
- [ ] Проверяется факт авторизации в Storyblocks.
- [ ] Детектируется невалидная или истекшая сессия.
- [ ] Пользователю предлагается ручной логин.
- [ ] Сессия сохраняется между перезапусками приложения.
- [ ] Состояние сессии доступно для UI и orchestration.

### 4.4 Flow ручного вмешательства

- [ ] Поддержано состояние `login required`.
- [ ] Поддержано состояние `challenge detected`.
- [ ] Есть явное ручное подтверждение пользователем после логина или challenge.
- [ ] Во время ожидания пользователь работает в уже открытом persistent browser profile.

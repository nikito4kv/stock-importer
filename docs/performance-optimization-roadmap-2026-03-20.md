# План Оптимизации Производительности

## Цель

Сделать приложение заметно быстрее и стабильнее без потери текущей бизнес-логики:

- быстрый импорт и старт проекта;
- предсказуемый и масштабируемый media-run;
- отзывчивый UI во время длительных запусков;
- минимальный технический долг в runtime-пути.

## Ключевые Принципы

- Сначала измеряем, потом оптимизируем.
- Оптимизируем горячие пути, а не весь код сразу.
- Убираем дубли и split-brain между new-core и legacy.
- Сохраняем совместимость форматов данных и сценариев пользователя.

## Текущие Узкие Места

- Смешение нового слоя и `legacy_core` в runtime-пути.
- Избыточный I/O: частые сохранения manifest/run и частый UI polling.
- Нечеткая конкурентная модель: часть настроек конкурентности не соответствует реальным пулам.
- Риски многопоточности для Storyblocks сессии.
- Алгоритмические потери: линейные поиски и повторный rebuild deduper.
- Тяжелые legacy-скрипты в корне репозитория, усложняющие поддержку и тесты.

## KPI И SLO (целевые)

- Время импорта проекта (DOCX -> project) <= 2 сек на 100 абзацев.
- Среднее время обработки абзаца в run снизить минимум на 30%.
- UI refresh latency <= 150 мс в активном run.
- Доля пустых/безрезультатных поисков без таймаута > 95%.
- Снижение объема файловых операций в run-path минимум на 50%.

## План Работ По Фазам

## Фаза 0: Наблюдаемость И База Метрик (1-2 дня)

- Ввести профилирование hot path: ingest, intent extraction, provider search, download, persist.
- Добавить counters/тайминги в event payload и отдельный perf-log.
- Зафиксировать baseline: median/p95 по каждому этапу на тестовом наборе сценариев.
- Вынести “перформанс-репорты” в `docs/benchmarks/`.

## Фаза 1: Быстрые Победы (3-5 дней)

- Ограничить Storyblocks путь до `paragraph_workers=1` (жесткий guard на запуске).
- Уменьшить частоту UI polling и переключить на incremental/live snapshot.
- Перейти на batched persist manifest/run по checkpoint вместо частых write-after-step.
- Оптимизировать event logging: буферизация и batch flush вместо открытия файла на каждый event.
- Добавить индекс `paragraph_no -> entry` в manifest-обработке, убрать линейный поиск в цикле.
- Перевести deduper на инкрементальное состояние run-level, убрать пересборку на каждый абзац.

Ожидаемый эффект: быстрое снижение I/O и CPU overhead без сложной миграции.

## Фаза 2: Конкурентность И Пайплайн (1-2 недели)

- Разделить конкурентность по режимам:
- Storyblocks mode: безопасный single-thread execution.
- Free-image mode: параллельные provider/query/download worker pools.
- Реально задействовать `provider_workers/download_workers/relevance_workers` через явные executors.
- Добавить backpressure на результаты поиска и очередь скачивания с лимитами.
- Сделать early-stop более агрессивным при достижении quality threshold.

Ожидаемый эффект: рост throughput run без ошибок thread ownership.

## Фаза 3: Оптимизация Кэшей И Сети (1 неделя)

- Перевести SQLite cache на persistent connection + WAL + подготовленные statements.
- Добавить TTL/версирование cache key для безопасной инвалидации.
- Ввести connection pooling для HTTP в free-image клиентах.
- Снизить повторные сетевые обращения за счет кэша provider query plan.
- Унифицировать retry/backoff policy для внешних запросов.

Ожидаемый эффект: меньше latency и дерганий сети, стабильнее массовые run.

## Фаза 4: Удаление Мертвого И Лишнего Кода (1-2 недели)

- Вынести `image_fetcher.py`, `video_fetcher.py`, `keyword_extractor.py` в `legacy_cli/`.
- Оставить в корне только thin wrappers (или полностью убрать из runtime-поставки).
- Формализовать adapter boundary: новые модули не импортируют `legacy_core` напрямую.
- Постепенно переносить функции из `legacy_core` в `pipeline/providers/services` по migration backlog.
- Удалить неиспользуемые настройки и поля, которые только сериализуются, но не влияют на поведение.

Ожидаемый эффект: проще сопровождение, меньше шанс регрессий, быстрее изменения.

## Фаза 5: Документация, Тесты, Гейты (3-5 дней)

- Починить drift между тестами и текущей структурой документации.
- Добавить performance regression tests для hot-path сценариев.
- В CI добавить отдельный perf-smoke job с порогами деградации.
- Обновить архитектурные правила: canonical runtime path и запрет обходов.

## Рекомендации По Стеку

- Не делать полный rewrite и не переводить весь проект на async сразу.
- Оставить основной orchestration в threads + селективный async/parallel только для I/O-heavy free-image path.
- Если Tk fallback не критичен для бизнеса, оставить одну UI платформу (PySide6), сократив дублирование.
- Изолировать или отключить по умолчанию хрупкие источники (generic web image) до улучшения качества/стабильности.

## Что Делать Нельзя

- Полный “big-bang” рефактор без промежуточных метрик.
- Удаление `legacy_core` до появления полной adapter-замены и regression coverage.
- Повышение параллелизма Storyblocks без thread-safe session модели.
- Массовые изменения форматов persistence без миграционной совместимости.

## Definition Of Done

- KPI из раздела SLO достигнуты на реальных сценариях.
- Нет деградации бизнес-логики и пользовательских флоу.
- Runtime путь очищен от лишнего/мертвого кода.
- Архитектурные границы зафиксированы в коде, тестах и документации.

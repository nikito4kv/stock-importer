# Фаза 5: Гейты Качества И Защита От Регрессий

Цель фазы: закрепить достигнутую производительность и не допускать отката.

## P5-01 [ ] Добавить performance regression tests

Действия:
1. Определить 3-5 каноничных performance-сценариев.
2. Добавить автозапуск сценариев с сохранением метрик.
3. Ввести пороги деградации по p50/p95 для ключевых этапов.

Артефакты:
- `tests/perf/` (новый набор)
- `docs/benchmarks/thresholds.md`

Проверка:
1. PR с деградацией выше порога падает на perf-гейте.
2. Порог и методика прозрачны и воспроизводимы.

## P5-02 [ ] Добавить CI job `perf-smoke`

Действия:
1. В CI workflow добавить отдельный job для быстрой perf-проверки.
2. Ограничить длительность job и размер набора данных.
3. Публиковать артефакт с метриками в CI.

Артефакты:
- `.github/workflows/ci.yml` (или отдельный workflow)

Проверка:
1. `perf-smoke` запускается на PR.
2. В артефактах CI есть отчет по метрикам.

## P5-03 [ ] Ввести архитектурный тест на запрет новых legacy импортов

Действия:
1. Добавить тест, сканирующий импорты new-core модулей.
2. Разрешить `legacy_core` только внутри `legacy_adapters/`.
3. Падать с понятным сообщением и списком нарушений.

Артефакты:
- `tests/test_phase2_architecture.py` или новый `tests/test_arch_imports.py`

Проверка:
1. Нарушение правила воспроизводимо ловится CI.
2. Список исключений минимален и явный.

## P5-04 [ ] Зафиксировать Definition of Done для performance-задач

Действия:
1. Создать шаблон PR для performance-изменений.
2. Включить обязательные пункты: baseline, after, риск, rollback plan.
3. Добавить checklist в CONTRIBUTING.

Артефакты:
- `.github/pull_request_template.md`
- `CONTRIBUTING.md`

Проверка:
1. Каждый perf PR содержит сравнение "до/после".
2. Команда использует единый формат ревью.

## P5-05 [ ] Добавить nightly benchmark job

Действия:
1. Настроить nightly workflow на расширенный набор сценариев.
2. Сохранять историю метрик (artifact или отдельный отчет в docs/benchmarks).
3. Добавить алерт при устойчивой деградации.

Артефакты:
- `.github/workflows/nightly-bench.yml`
- `docs/benchmarks/history/`

Проверка:
1. Ежедневно появляется отчет с метриками.
2. Тренд деградации виден без ручного запуска.

## P5-06 [ ] Закрыть тестовые пробелы по оптимизированным зонам

Действия:
1. Добавить тесты на batched persist и recovery после crash.
2. Добавить тесты на корректность polling/live snapshot после оптимизации.
3. Добавить тесты на конкурентные сценарии free-image режима.

Артефакты:
- `tests/test_media_pipeline.py`
- `tests/test_ui_controller.py`
- `tests/test_phase9_reliability.py`

Проверка:
1. Новые оптимизации покрыты regression-тестами.
2. Нет незакрытых критичных путей без тестов.

## P5-07 [ ] Подготовить релизный performance-отчет

Действия:
1. Сравнить baseline Фазы 0 и финальные результаты после Фазы 5.
2. Зафиксировать выигрыш по каждому KPI.
3. Описать оставшиеся ограничения и следующий backlog.

Артефакты:
- `docs/benchmarks/final-performance-report-2026-XX-XX.md`

Проверка:
1. Есть подтвержденные цифры улучшения.
2. Отчет пригоден как вход для следующего цикла оптимизаций.

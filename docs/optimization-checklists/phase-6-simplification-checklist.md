# Упрощение приложения: чеклист

Примечание: это roadmap-документ phase 6. Упоминания удаленных функций описывают
область cleanup и не являются текущим пользовательским контрактом.

Цель этого списка:
- убрать лишнюю сложность;
- оставить только то, что реально нужно для повседневной работы;
- сократить количество режимов, скрытой логики и редких сценариев.

Главная идея:
- desktop UI поддерживается только через `Qt`/`PySide6`;
- приложение должно быть простым инструментом для Storyblocks;
- бесплатные источники нужны только как запасной вариант для картинок;
- управление запуском должно быть понятным и коротким;
- лишние уровни абстракции нужно убрать.

## Что хотим получить в итоге

- Один поддерживаемый desktop UI слой на `PySide6`.
- Один основной сценарий работы:
  - видео из Storyblocks;
  - картинки из Storyblocks;
  - если надо, запасные картинки из бесплатных источников.
- Минимум сложных настроек в интерфейсе.
- Минимум редких режимов и скрытой логики.
- Меньше кода, который поддерживает малоиспользуемые функции.

## P6-00 [ ] Убрать Tk-слой и оставить один desktop stack

Подробное ТЗ:
[task-plans/p6-00.md](./task-plans/p6-00.md)

Анализ:
Поддержка двух толстых desktop-веток раздувает каждую UI-задачу и скрывает
дефекты `Qt` за fallback-логикой.

Что упростить:
- удалить `ui/tk_app.py`;
- убрать fallback `Qt -> Tk`;
- зафиксировать `PySide6` как единственный поддерживаемый desktop UI;
- оставить `desktop_stack` только как legacy-совместимое поле persisted settings.

Шаги:
- [ ] Переписать bootstrap desktop UI на единственный `Qt` path.
- [ ] Удалить `ui/tk_app.py` и связанные exports/tests.
- [ ] Нормализовать загрузку старых settings без возврата к `Tk`.
- [ ] Обновить task-plan'ы и обзорные документы под `Qt-only` flow.

Проверка:
- [ ] В runtime нет fallback на `Tk`.
- [ ] `ui/tk_app.py` удален из production-кода.
- [ ] Старый `settings.json` с `desktop_stack="tk"` не ломает запуск и не меняет выбранный UI.

## P6-01 [ ] Упростить систему провайдеров

Анализ:
Сейчас приложение устроено как универсальная система для большого числа провайдеров. Для реальной работы это не нужно.

Оставить:
- Storyblocks для видео.
- Storyblocks для изображений.
- 3 бесплатных источника для запасного варианта:
  - Openverse
  - Pixabay
  - Pexels

Убрать:
- generic web image path;
- Bing;
- лишние provider-specific ветки, которые не нужны для этих 4 источников.

Шаги:
- [ ] Удалить Bing из runtime-конфига, реестра провайдеров и UI.
- [ ] Удалить generic web image path и связанные с ним ветки.
- [ ] Упростить provider registry под фиксированный набор провайдеров.
- [ ] Убрать код, который нужен только для редко используемых или больше не поддерживаемых image providers.
- [ ] Проверить, что Storyblocks и 3 бесплатных image providers собираются и работают без старой общей "платформенной" логики.

Проверка:
- [ ] В приложении доступны только Storyblocks + Openverse/Pixabay/Pexels.
- [ ] Нет больше выбора Bing/generic web image path.
- [ ] Поиск и скачивание продолжают работать для оставшихся источников.

## P6-02 [ ] Убрать сложную логику приоритетов и стратегий выбора

Анализ:
Сейчас в приложении много логики про группы провайдеров, приоритеты и стратегии выбора. Это полезно для конструктора платформы, но не для обычного приложения.

Что упростить:
- убрать сложную систему `priority`;
- убрать сложную систему `provider_group`;
- убрать многоступенчатые "стратегии выбора";
- оставить прямую, понятную схему:
  - сначала Storyblocks;
  - потом бесплатные картинки как запасной вариант.

Шаги:
- [ ] Упростить runtime wiring так, чтобы выбор источников был прямым и явным.
- [ ] Убрать лишние ветки `resolve_image_strategy` и похожую косвенную логику.
- [ ] Убрать настройки и код, которые существуют только для перестановки приоритетов между провайдерами.
- [ ] Проверить, что продуктовая логика легко читается без знания внутренних "групп" и "стратегий".

Проверка:
- [ ] Порядок выбора источников понятен из кода без дополнительной расшифровки.
- [ ] Storyblocks идет первым.
- [ ] Бесплатные image providers работают только как fallback.

## P6-03 [ ] Упростить настройки в интерфейсе

Подробное ТЗ: [task-plans/p6-03.md](./task-plans/p6-03.md)

Анализ:
Сейчас в UI слишком много технических ручек. Для обычного пользователя они лишние.

Убрать из обычного интерфейса:
- количество потоков;
- размеры очередей;
- лимиты;
- таймауты;
- retry budget.

Оставить вместо этого:
- режим `Обычный`;
- режим `Быстрый`;
- режим `Custom`.

Для `Custom` оставить только то, что реально нужно пользователю:
- возможность вручную настроить задержки/таймауты, если это нужно для скачивания.

Шаги:
- [ ] Убрать из основного UI технические поля, которые не нужны большинству пользователей.
- [ ] Ввести 2 простых готовых режима: `Обычный` и `Быстрый`.
- [ ] Оставить `Custom` как отдельный режим для ручной настройки.
- [ ] Свести `Custom` к небольшому набору действительно полезных параметров.
- [ ] Удалить настройки, которые больше не используются после упрощения pipeline.

Проверка:
- [ ] Основной экран запуска стал короче и понятнее.
- [ ] Пользователь может запустить работу без знания потоков, очередей и retry policy.
- [ ] `Custom` по-прежнему позволяет вручную настроить задержки скачивания.

## P6-04 [ ] Упростить управление запуском

Анализ:
Пауза, продолжение, повтор выбранных абзацев и live journal сильно раздувают код.

Оставить только:
- `Start`
- `Cancel`
- `Rerun full run`

Убрать:
- pause;
- resume;
- retry selected;
- rerun subsets;
- live journal;
- лишнюю checkpoint-логику, если она нужна только для этих функций.

Шаги:
- [ ] Упростить `pipeline/orchestrator.py` под короткий набор команд.
- [ ] Упростить `ui/controller.py` и убрать кнопки/состояния для pause/resume и частичных rerun.
- [ ] Удалить код и тесты, которые существуют только ради pause/resume и выборочного перезапуска.
- [ ] Свести состояние запуска к простым статусам: running / cancelled / completed / failed.

Проверка:
- [ ] В UI остались только Start, Cancel и Rerun full run.
- [ ] В коде больше нет отдельной сложной логики pause/resume.
- [ ] Статусы прогона стали проще и понятнее.

## P6-05 [ ] Упростить работу с браузером

Анализ:
Если используется только один Storyblocks-профиль, сложная система профилей не нужна.

Что оставить:
- один рабочий Storyblocks-профиль.

Что убрать или сократить:
- поддержку нескольких профилей;
- лишние сценарии импорта профилей;
- сложные ветки, которые нужны только для multi-profile UX.

Шаги:
- [ ] Упростить browser/session слой под один постоянный Storyblocks-профиль.
- [ ] Убрать из UI выбор и управление несколькими профилями, если это не нужно в реальном использовании.
- [ ] Упростить storage и runtime-логику, связанную с несколькими профилями.
- [ ] Проверить shutdown/startup lifecycle после упрощения.
- [ ] Убедиться, что smoke output показывает один `storyblocks_profile_id`, а shutdown закрывает native login browser.

Проверка:
- [ ] В приложении используется один Storyblocks-профиль.
- [ ] Код session/profile управления стал заметно проще.
- [ ] Авторизация и работа Storyblocks не ухудшились.
- [ ] Во вкладке `Сессия` нет действий для переключения или очистки managed профилей.

## P6-06 [ ] Максимально упростить поиск картинок

Анализ:
Сейчас поиск картинок пытается быть "умным", но это создает много лишней логики.

Что хотим:
- 1-2 понятных шаблона запроса;
- без сложных provider-specific rewrite;
- без сложной проверки качества;
- просто брать первые подходящие результаты.

Допущение для этого этапа:
- проверку качества картинок можно убрать;
- не пытаться ранжировать результаты слишком умно;
- брать первые попавшиеся изображения из оставшихся источников.

Шаги:
- [ ] Упростить query planner для image providers.
- [ ] Убрать сложные provider-specific query rewrite.
- [ ] Убрать или сильно сократить фильтрацию/оценку качества изображений.
- [ ] Упростить ranking image candidates до минимально нужного поведения.
- [ ] Проверить, что результат все еще предсказуем и не ломает основной пользовательский сценарий.

Проверка:
- [ ] Поиск картинок работает через 1-2 простых шаблона.
- [ ] В коде больше нет тяжелой логики "умного" выбора картинок.
- [ ] Изображения выбираются проще и быстрее.

## P6-08 [ ] Сократить portable/release слой, но оставить переносимый bundle

Анализ:
Полный release-слой для широкой раздачи не нужен, но переносимый bundle нужен, если он используется для переноса на сервер и последующей установки.

Решение:
- не вырезать bundle полностью;
- оставить минимальный переносимый export/install path;
- убрать только лишнюю release-сложность, которая не нужна для реального деплоя.

Что оставить:
- bundle/export, который можно перенести на сервер;
- установку/распаковку на целевой машине.

Что можно убрать:
- лишние release-сценарии;
- лишнюю packaging-обвязку для широкого распространения;
- лишние тесты и документы, если они существуют только ради "дистрибутива для всех".

Шаги:
- [ ] Определить минимальный набор файлов и скриптов для переноса на сервер.
- [ ] Оставить только тот bundle path, который реально используется.
- [ ] Удалить лишнюю release-обвязку, не связанную с переносом и установкой.
- [ ] Обновить документацию под упрощенный deploy path.

Проверка:
- [ ] Bundle по-прежнему можно перенести на сервер.
- [ ] Установка/распаковка на целевой машине остается рабочей.
- [ ] Вокруг этого сценария стало меньше лишнего кода и документов.

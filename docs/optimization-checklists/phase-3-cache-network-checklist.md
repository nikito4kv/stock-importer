# Фаза 3: Кэш И Сеть

Цель фазы: снизить сетевую latency и overhead кэшей в provider pipeline.

## P3-01 [ ] Перевести SQLite cache на persistent connection

Действия:
1. Убрать открытие нового sqlite соединения на каждый `get/set`.
2. Держать одно соединение на инстанс кэша с корректным закрытием.
3. Защитить доступ lock-ом на запись и чтение.

Артефакты:
- `providers/images/caching.py`

Проверка:
1. В профиле вызовов нет массового churn sqlite connect/disconnect.
2. Функциональность кэша не изменилась.

## P3-02 [ ] Включить WAL и безопасные PRAGMA

Действия:
1. Включить `journal_mode=WAL`.
2. Настроить `synchronous` и `busy_timeout`.
3. Проверить стабильность в многопоточном доступе.

Артефакты:
- `providers/images/caching.py`

Проверка:
1. Уменьшены блокировки БД под нагрузкой.
2. Нет потери данных при штатном завершении.

## P3-03 [ ] Добавить TTL к кэшу результатов

Действия:
1. Расширить схему кэша полем `created_at`.
2. На чтении отбрасывать устаревшие записи по TTL.
3. Добавить периодическую очистку протухших записей.

Артефакты:
- `providers/images/caching.py`
- `config/settings.py` (если TTL в конфиге)

Проверка:
1. Устаревшие данные не используются после истечения TTL.
2. Размер кэша не растет бесконечно.

## P3-04 [ ] Ввести версионирование cache key

Действия:
1. Добавить `cache_key_version` в ключи search/metadata cache.
2. Увеличивать версию при изменении логики ранжирования/фильтрации.
3. Документировать правило обновления версии.

Артефакты:
- `providers/images/caching.py`
- `providers/images/filtering.py`
- `docs/benchmarks/` (краткое правило)

Проверка:
1. После изменения алгоритма нет смешивания старых и новых результатов кэша.
2. Поведение предсказуемо при релизе.

## P3-05 [ ] Ускорить provider search за счет session reuse

Действия:
1. Ввести session reuse/pooling для HTTP клиентов free-image провайдеров.
2. Перенести повторяющиеся заголовки и таймауты в общий client config.
3. Добавить явное закрытие клиентов на shutdown.

Артефакты:
- `providers/images/clients.py`
- `providers/images/service.py`

Проверка:
1. Снижается latency последовательных запросов к одному провайдеру.
2. Нет утечек сокетов/дескрипторов.

## P3-06 [ ] Унифицировать retry/backoff policy

Действия:
1. Задать общий backoff-профиль для сетевых вызовов.
2. Разделить retryable и non-retryable ошибки.
3. Логировать `attempt_count` и финальный статус.

Артефакты:
- `legacy_core/retry.py` и/или новый shared retry модуль
- `providers/images/*`
- `pipeline/media.py`

Проверка:
1. Повторные попытки происходят только для транзиентных ошибок.
2. Нет каскадного повторения без ограничения.

## P3-07 [ ] Добавить тесты кэша и сетевых деградаций

Действия:
1. Добавить тесты TTL/версии кэша.
2. Добавить тесты retry/backoff на искусственных фейлах.
3. Добавить тесты конкурентного чтения/записи кэша.

Артефакты:
- `tests/test_image_provider_architecture.py`
- новые тесты в `tests/`

Проверка:
1. Тесты воспроизводимо покрывают кэшевые и сетевые сценарии.
2. Регрессии ловятся до merge.

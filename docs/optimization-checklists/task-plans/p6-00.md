# P6-00. ТЗ на удаление Tk UI слоя и переход на один desktop stack

Источник задачи: продуктовое решение по сокращению стоимости поддержки UI внутри phase 6.

## Цель

Убрать `Tk` как второй desktop-слой и оставить в продукте только `Qt`/`PySide6`.

Результат задачи:

- в приложении остается один поддерживаемый desktop UI: `PySide6`;
- `ui/tk_app.py` и Tk-specific wiring исчезают из runtime path;
- новые phase-6 UI-задачи больше не требуют дублировать изменения в двух GUI-ветках;
- fallback `Qt -> Tk` больше не маскирует реальные ошибки Qt path;
- документация и task-plan'ы дальше описывают один UI-слой, а не две параллельные реализации.

## Почему это нужно

Сейчас Tk-слой создает постоянную стоимость без продуктовой отдачи:

- любую UI-фичу приходится реализовывать и тестировать дважды;
- `ui/tk_app.py` уже отстает по структуре и накапливает drift;
- task-plan'ы P6-03, P6-04, P6-05 и P6-07 удваивают UI-объем только ради parity;
- fallback на Tk скрывает реальные дефекты `PySide6`, вместо того чтобы делать их видимыми сразу;
- выбранный desktop stack проекта уже зафиксирован как `PySide6`, поэтому поддержка второго толстого слоя не нужна.

## Границы задачи

Входит в P6-00:

- удаление `Tk` как поддерживаемого desktop UI;
- удаление runtime/import fallback на `ui/tk_app.py`;
- удаление Tk-экспортов, Tk-entrypoint'ов и Tk-specific docs/tests;
- выравнивание task-plan'ов под `Qt-only` trajectory;
- backward-compatible чтение старых настроек, где мог фигурировать `desktop_stack`.

Не входит в P6-00:

- редизайн Qt-экранов сам по себе; это делают следующие P6-задачи;
- переписывание controller/contracts только ради архитектурной красоты, если это не нужно для удаления Tk;
- смена `PySide6` на другой GUI framework;
- глубокая release-упаковка; это отдельный P6-08.

## Важное уточнение для текущего состояния ветки

Если P6-01, P6-02 и P6-03 уже реализованы, P6-00 должен выступать не как
абстрактный prerequisite, а как cleanup-задача поверх уже выполненной работы.

Это означает:

- не откатывать продуктовые изменения из P6-01/P6-02/P6-03;
- удалить именно Tk-ветку, которая дублирует уже внедренные упрощения;
- оставить `Qt` и shared controller/contracts как единственный поддерживаемый путь.

Что именно P6-00 должен подчистить поверх уже сделанных задач:

- из P6-01: Tk-реализацию сокращенного набора free-image providers и related labels/handlers;
- из P6-02: Tk-use-sites и тестовые/документационные следы, если они все еще держат старую dual-UI обязанность вокруг provider-order logic;
- из P6-03: Tk-реализацию `launch_profile`, `Custom` timing group, preset restore и прочих форменных веток, уже реализованных для Qt.

Практическое правило:

- shared логика в `ui/controller.py`, `ui/contracts.py`, `ui/presentation.py`,
  `ui/launch_profiles.py`, появившаяся ради P6-01/P6-02/P6-03, сохраняется, если
  она нужна `Qt`;
- удалять нужно только то, что остается жить исключительно ради Tk parity.

## Точное продуктовое решение

### 1. Один поддерживаемый desktop UI

После P6-00 продукт поддерживает только `Qt`/`PySide6`.

Практическое правило:

- если `PySide6` не установлен или не поднимается, приложение должно завершаться с явной ошибкой;
- fallback на `Tk` больше не считается допустимым поведением.

### 2. `desktop_stack` больше не управляет выбором UI

Поле `desktop_stack` можно оставить временно в persisted settings только как legacy payload, но не как живой продуктовый переключатель.

Минимально допустимое поведение:

- старый `settings.json` с `desktop_stack="tk"` не ломает загрузку;
- runtime все равно запускает `Qt` path;
- новое пользовательское поведение не обещает выбор между `Qt` и `Tk`.

### 3. Следующие phase-6 UI-задачи должны считать Tk уже удаленным

После появления P6-00 все планы по интерфейсу должны опираться только на:

- `ui/contracts.py`
- `ui/controller.py`
- `ui/presentation.py`
- `ui/qt_app.py`

`ui/tk_app.py` не должен фигурировать как обязательная часть новых работ.

## Где сейчас сидит лишняя логика

- Tk desktop implementation (historical, file removed): `ui/tk_app.py`
- Выбор desktop app и fallback logic: [ui/__init__.py](../../../ui/__init__.py)
- Persisted app settings с `desktop_stack`: [config/settings.py](../../../config/settings.py)
- Load/save `desktop_stack`: [storage/repositories.py](../../../storage/repositories.py)
- Runtime/tests, где проверяется общий desktop contract: [app/runtime.py](../../../app/runtime.py), [tests/test_phase2_architecture.py](../../../tests/test_phase2_architecture.py)
- Документация и task-plan'ы, где `Qt` и `Tk` описаны как равноправные ветки: [docs/optimization-checklists/task-plans/](./)

## Definition of Done

- В кодовой базе нет живого runtime path, который запускает `Tk` UI.
- `ui/tk_app.py` удален из production-кода.
- `ui/__init__.py` больше не экспортирует `DesktopTkApp` и не делает fallback на Tk.
- Старый `settings.json` с `desktop_stack != "pyside6"` читается без падения, но не возвращает продукт к Tk.
- Документация и phase-6 task-plan'ы больше не требуют parity между `Qt` и `Tk`.
- Все последующие UI-задачи описаны как изменения одного desktop-слоя.

## Точный состав изменений по файлам

### Runtime и UI bootstrap

Изменить:

- [ui/__init__.py](../../../ui/__init__.py)
- [app/runtime.py](../../../app/runtime.py)
- при необходимости [app/__main__.py](../../../app/__main__.py)

Что сделать:

- убрать импорт и запуск `DesktopTkApp`;
- убрать fallback-логику `Qt -> Tk`;
- оставить один явный путь запуска `Qt`;
- сделать ошибку отсутствия `PySide6` явной и понятной.

Что должно получиться:

- runtime больше не скрывает Qt-проблемы за fallback-веткой;
- startup contract становится проще и честнее.

Как проверить:

- поиск по `ui/__init__.py` не находит `launch_tk_app` и `DesktopTkApp`;
- desktop launch path ведет только в `ui/qt_app.py`.

### Tk слой

Изменить:

- исторический файл `ui/tk_app.py` (удален из продукта)

Что сделать:

- удалить файл из поддерживаемого продукта после того, как все импорты и runtime references убраны;
- вместе с файлом удалить Tk-реализацию уже выполненных phase-6 изменений:
  - provider selectors и related handlers из P6-01;
  - use-sites provider-order/source-selection UI из P6-02, если они не нужны вне Tk;
  - `launch_profile`, `Custom` timing, preset restore и связанные form bindings из P6-03;
- не оставлять "мертвый" fallback-код как архив внутри `ui/`.

Что должно получиться:

- в репозитории больше нет второго GUI-адаптера, который нужно синхронизировать вручную.

Как проверить:

- `glob`/`grep` по `ui/` не находят `tk_app.py`, `DesktopTkApp`, `launch_tk_app`.

### Настройки и backward compatibility

Изменить:

- [config/settings.py](../../../config/settings.py)
- [storage/repositories.py](../../../storage/repositories.py)

Что сделать:

- при желании оставить поле как legacy-read-compatible, но не как пользовательскую настройку.

Что должно получиться:

Что должно получиться:

Как проверить:

### Документация и планы

Изменить:

- при необходимости [ui/README.md](../../../ui/README.md) и смежные обзорные документы.

## Атомарный план реализации

### Шаг 1. Зафиксировать `Qt-only` support policy

Что сделать:

- Явно принять, что `PySide6` - единственный поддерживаемый desktop stack.
- Зафиксировать, что fallback на Tk больше не поддерживается.
- Добавить этот план как prerequisite для остальных UI-задач phase 6.

Что должно получиться:

- у команды появляется один простой support contract;
- последующие планы больше не умножают UI-объем на два.

Как проверить:

- в task-plan'ах нет двусмысленности, нужно ли продолжать feature parity для Tk.

### Шаг 2. Убрать Tk из runtime/bootstrap

Что сделать:

- переписать `ui/__init__.py` на один `Qt` path;
- убрать `except`/fallback-ветки, которые вели в Tk;
- сделать ошибки загрузки Qt явными.

Что должно получиться:

- любые дефекты Qt path становятся видимыми сразу;
- приложение больше не переключается молча на другой UI.

Как проверить:

- поиск по `ui/__init__.py` не находит Tk fallback identifiers;
- запуск desktop app идет только через `Qt`.

### Шаг 3. Удалить `ui/tk_app.py` и связанные экспорты

Что сделать:

- удалить `ui/tk_app.py` после очистки импортов;
- убрать Tk из `__all__`, helper'ов и прочих re-export'ов.
- считать частью этого шага зачистку уже реализованных Tk-веток из P6-01/P6-02/P6-03:
  - provider UI и handlers после P6-01;
  - Tk-consumers упрощенного provider-order/snapshot contract после P6-02;
  - Tk-form bindings для `launch_profile` и `Custom` после P6-03.

Что должно получиться:

- в `ui/` остается одна толстая desktop-реализация.
- уже выполненные продуктовые упрощения из P6-01/P6-02/P6-03 продолжают жить только в `Qt`, без второй GUI-ветки.

Как проверить:

- `grep` по коду не находит `DesktopTkApp`, `launch_tk_app`, `tkinter`.

### Шаг 4. Нормализовать legacy settings и tests

Что сделать:

- сделать безопасную загрузку старого `desktop_stack` без возврата Tk;
- удалить тесты и code paths, где dual-stack является частью контракта.
- после удаления `ui/tk_app.py` пройтись по shared файлам и убрать мертвые ссылки,
  оставшиеся после уже выполненных P6-01/P6-02/P6-03, если они существовали
  только ради Tk adapter:
  - неиспользуемые imports;
  - helper branches для второго GUI-слоя;
  - docs/test wording про parity между `Qt` и `Tk`.

Что должно получиться:

- старые данные читаются;
- новый runtime contract остается `Qt-only`.
- кодовая база больше не содержит следов того, что P6-01/P6-02/P6-03 когда-то
  обязаны были поддерживаться в двух UI-ветках.

Как проверить:

- regression test на legacy settings проходит;
- тесты не зависят от Tk fallback.
- статический поиск не находит мертвых references, оставшихся после удаления Tk.

### Шаг 5. Переписать phase-6 task-plan'ы под один UI-слой

Что сделать:

- убрать `Tk` как обязательную цель из P6-01, P6-03, P6-04, P6-05, P6-07;
- заменить парные шаги `Qt + Tk` на один Qt-шаг;
- при необходимости добавить note, что P6-00 - prerequisite.

Что должно получиться:

- оставшиеся phase-6 планы больше не дублируют одну и ту же UI-работу.

Как проверить:

- в task-plan'ах не осталось acceptance criteria уровня "Qt и Tk используют одинаковый контракт".

### Шаг 6. Финальная верификация

Что сделать:

- прогнать статический поиск по Tk identifiers;
- прогнать таргетные тесты;
- сделать smoke запуска Qt-приложения.

Команды проверки:

- `ruff check .`
- `python -m unittest tests.test_phase2_architecture`
- `python -m unittest discover -s tests`
- `Get-ChildItem app,config,docs,storage,tests,ui -Recurse -File | Select-String -Pattern 'DesktopTkApp|launch_tk_app|tkinter|ui/tk_app.py'`

Ручная smoke-проверка:

- запустить `python -m app`;
- убедиться, что стартует только Qt UI;
- при проблеме с Qt получить явную ошибку, а не молчаливый переход на Tk.

## Acceptance criteria

- Tk больше не является частью поддерживаемого desktop product path;
- новые UI-задачи phase 6 не требуют изменений в `ui/tk_app.py`;
- cost любой следующей UI-фичи уменьшается, потому что остается один GUI-слой;
- старые настройки не ломают запуск после отказа от Tk;
- документация и task-plan'ы совпадают с новым `Qt-only` решением.

## Практические советы

- Делайте этот план первым среди phase-6 UI-задач. Иначе P6-03/P6-04/P6-05/P6-07 снова раздуют объём работ вдвое.
- Если P6-01/P6-02/P6-03 уже сделаны, выполняйте P6-00 следующим cleanup-шагом до P6-04/P6-05/P6-07, чтобы не тянуть старую цену dual-UI дальше.
- Не оставляйте Tk как "временный скрытый fallback". Такой код продолжит жить в тестах, review-комментариях и будущих фичах.
- Не превращайте `desktop_stack` в условный переключатель с одним вариантом. Если поле остается, оно должно быть только legacy-compatible, а не продуктовым выбором.
- Сначала уберите runtime/import references, потом удаляйте файл `ui/tk_app.py`. Иначе легко сломать imports и получить частичный cleanup.

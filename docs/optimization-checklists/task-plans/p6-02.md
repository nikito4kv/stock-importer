# P6-02. Детальное ТЗ: убрать сложную логику приоритетов и стратегий выбора

Источник задачи: [phase-6-simplification-checklist.md](./phase-6-simplification-checklist.md)

## Цель

Сделать выбор image sources прямым, читаемым и продуктовым:

- если в режиме проекта разрешены изображения Storyblocks, сначала ищется `storyblocks_image`;
- бесплатные image providers запускаются только после Storyblocks и используются только как fallback;
- порядок поиска должен быть понятен из кода без знания `priority`, `provider_group` и скрытых "strategy" helper-ов;
- код выбора источников должен собираться из явных правил режима проекта, а не из нескольких слоев косвенной логики.

Ожидаемый итог после P6-02:

- разработчик открывает [pipeline/media.py](../../pipeline/media.py) и сразу видит порядок поиска;
- registry перестает быть "движком стратегии" и остается каталогом провайдеров;
- настройки и snapshot-поля, которые нужны только для перестановки image-provider приоритетов, исчезают;
- mixed image behavior читается как "Storyblocks first, free fallback", а не как комбинация нескольких флагов и сортировок.

## Важное уточнение по продуктовой логике

В исходном пункте плана есть скрытый конфликт: в приложении существует режим `free_images_only` из [domain/project_modes.py](../../domain/project_modes.py), а формулировка "бесплатные image providers работают только как fallback" буквально несовместима с режимом, где Storyblocks вообще не участвует.

Для выполнимого ТЗ принимается следующее решение:

- во всех режимах, где включен `storyblocks_image`, бесплатные image providers работают только как fallback;
- режим `free_images_only` не удаляется в P6-02, а остается явным отдельным исключением;
- это исключение должно быть реализовано прямо и локально, без общей strategy-системы;
- если продукт хочет полностью буквальное правило "free images только fallback без исключений", это уже отдельная задача на удаление `free_images_only`, а не часть P6-02.

Иначе исполнитель либо сломает существующий режим, либо напишет ТЗ, которое невозможно выполнить честно.

## Проблема в текущей реализации

Сейчас логика выбора источников размазана сразу по нескольким слоям:

- [providers/registry.py](../../providers/registry.py) сортирует providers через `priority`, делит их на `provider_group` и возвращает `resolve_image_strategy(...)`;
- [pipeline/media.py](../../pipeline/media.py) еще раз интерпретирует результат стратегии и поверх него накладывает `storyblocks_images_enabled`, `free_images_enabled`, `mixed_image_fallback`, `free_images_only`;
- [providers/images/clients.py](../../providers/images/clients.py) повторно сортирует free-image providers по `descriptor.priority`;
- [providers/images/querying.py](../../providers/images/querying.py) меняет query rewrite по `provider_group`;
- [providers/images/filtering.py](../../providers/images/filtering.py) добавляет score bonus/penalty по `priority` и `provider_group`;
- [config/settings.py](../../config/settings.py), [storage/repositories.py](../../storage/repositories.py) и [ui/controller.py](../../ui/controller.py) хранят и прокидывают поля `image_provider_priority`, `mixed_image_fallback`, `free_images_only`, которые влияют на порядок выбора косвенно.

Из-за этого один и тот же продуктовый вопрос "в каком порядке ищутся картинки?" приходится восстанавливать из нескольких мест сразу:

- [providers/registry.py](../../providers/registry.py)
- [pipeline/media.py](../../pipeline/media.py)
- [ui/controller.py](../../ui/controller.py)
- [storage/repositories.py](../../storage/repositories.py)

Это и есть главный дефект, который должен устранять P6-02.

## Точное продуктовое решение

### 1. Источник истины для состава провайдеров

Состав доступных providers по-прежнему определяется режимом проекта в [domain/project_modes.py](../../domain/project_modes.py) и helper-ом `provider_ids_for_mode(...)`.

P6-02 не должен создавать вторую параллельную матрицу режимов.

### 2. Явный порядок image sources

Для image selection должен существовать один прямой контракт:

- `storyblocks_image` относится к primary image path;
- `pexels`, `pixabay`, `openverse` и другие оставшиеся free-image providers относятся к fallback image path;
- если `storyblocks_images_enabled=False`, primary image path пуст;
- если `free_images_enabled=False`, fallback image path пуст;
- если оба пути включены, сначала выполняется Storyblocks path, потом fallback path;
- free providers не должны заполнять `supporting_assets`, пока доступен Storyblocks path;
- `free_images_only` разрешен только как отдельный явно описанный special-case.

### 3. Явный порядок внутри free-image path

Порядок бесплатных провайдеров должен задаваться одним явным ordered list, а не суммой:

- `image_provider_priority` из settings;
- `priority` в registry;
- повторной сортировки в `providers/images/clients.py`.

После P6-02 должен существовать один ответ на вопрос "в каком порядке перебираются free-image providers?".

### 4. Safety и concurrency не переписываются

P6-02 не отменяет:

- `resolve_concurrency_mode(...)` как safety-правило для Storyblocks;
- serial paragraph guard в [ui/controller.py](../../ui/controller.py);
- ограничения Storyblocks browser/session path;
- product modes в [domain/project_modes.py](../../domain/project_modes.py).

Нужно упростить only provider choice logic, а не весь media pipeline целиком.

## Зависимость от P6-01

Если P6-01 еще не выполнен, P6-02 все равно можно делать, но это усложнит diff.

Практически правильный порядок:

1. сначала убрать лишние providers и generic-web path по P6-01;
2. потом упрощать `priority` / `provider_group` / strategy wiring по P6-02;
3. затем чистить UI knobs по P6-03.

Причина простая: иначе в P6-02 приходится одновременно решать две разные задачи:

- какие providers вообще остаются в продукте;
- как упрощать порядок их выбора.

Это разные уровни изменений, и смешивать их в одну реализацию рискованно.

## Границы задачи

Входит в P6-02:

- упрощение registry и image selection contract;
- удаление косвенной strategy-логики;
- удаление settings-полей, которые существуют только ради перестановки image-provider order;
- упрощение query-planning/ranking веток, если они завязаны на `provider_group`/`priority`;
- обновление manifest payload и тестов под новый контракт;
- документация по новому порядку выбора источников.

Не входит в P6-02:

- удаление `bing`, `wikimedia`, `generic_web_image`, если это не сделано в P6-01;
- упрощение UI technical knobs как пользовательского сценария, это уже P6-03;
- переделка Storyblocks auth/session/browser automation;
- переписывание всего `legacy_core`;
- удаление `free_images_only` как продуктового режима.

## Definition of Done

- В runtime-коде больше нет `resolve_image_strategy(...)`.
- Порядок image source selection читается из одного явного участка кода.
- `storyblocks_image` всегда идет первым, если этот путь включен.
- Free-image providers не участвуют в primary Storyblocks path.
- В mixed режимах free-image providers заполняют только fallback branch.
- `image_provider_priority` и `mixed_image_fallback` больше не участвуют в новом runtime contract.
- Если `provider_group` и `priority` физически еще существуют как поля на промежуточном шаге, production logic больше не должна на них опираться.
- Старые `settings.json` и preset payload не ломают приложение после изменения contract.
- Тесты описывают прямой продуктовый порядок выбора, а не старую strategy-архитектуру.

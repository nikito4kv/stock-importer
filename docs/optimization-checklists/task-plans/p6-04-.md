# P6-04. ТЗ на упрощение управления запуском

Исходный пункт плана: [phase-6-simplification-checklist.md](./phase-6-simplification-checklist.md)

Предварительное условие: [p6-00-qt-only-ui-layer-spec-2026-03-21.md](./p6-00-qt-only-ui-layer-spec-2026-03-21.md) уже выполнен, Tk-слой удален.

## Цель

Свести управление запуском к короткому, понятному и устойчивому контракту:

- `Start`
- `Cancel`
- `Rerun full run`

Нужно убрать из продукта и кода все механики, которые существуют только ради
промежуточного управления прогоном:

- `pause`
- `resume`
- `retry failed`
- `rerun current paragraph`
- `rerun selected paragraphs`
- live journal в состоянии UI
- checkpoint-логику, если она нужна только для pause/resume и частичных rerun

После P6-04 запуск должен восприниматься как простой жизненный цикл:

- run стартует;
- либо доходит до `completed`/`failed`;
- либо останавливается через `cancelled`;
- при необходимости пользователь запускает новый полный прогон целиком.

## Важные уточнения по продуктовому контракту

- `Start` может и дальше учитывать предварительно выбранный subset абзацев, если это уже существующий сценарий создания нового run. P6-04 не обязан убирать старт по выбранным абзацам; он убирает только pause/resume и частичные rerun уже созданного run.
- `Rerun full run` всегда игнорирует `selected_paragraphs` исходного запуска и текущий selection в форме. Это отдельное создание нового полного run, а не "повтор того, что раньше запускали".
- Детальные `RunStage` можно оставить. Упрощение касается жизненного цикла run и управляющих действий, а не полезных технических стадий вроде `provider_search`, `download`, `relevance`, `persist`.
- Event bus и recorder можно оставить как инфраструктуру для логирования и диагностики, но live journal не должен больше жить в run state, polling и desktop UI.

## Границы задачи

Входит в P6-04:

- упрощение `RunStatus` и модели `Run`;
- упрощение `RunOrchestrator`;
- упрощение `ParagraphMediaRunService` и runtime API;
- удаление pause/resume/partial rerun из `ui/controller.py`;
- удаление live journal из live snapshot/state/UI;
- упрощение polling под новый набор terminal statuses;
- удаление или переписывание тестов, которые существуют только ради pause/resume и частичного rerun;
- сохранение совместимости со старыми `run.json`, если в workspace уже лежат старые статусы и payload.

Не входит в P6-04:

- полная переделка `RunStage`: детальные стадии (`provider_search`, `download`,
  `relevance`, `persist`) можно оставить;
- переделка выбора абзацев до старта run: пользователь по-прежнему может
  стартовать новый run не на всём проекте, если это уже часть продукта;
- переделка event bus и event recorder как инфраструктуры;
- удаление лог-экспорта как отдельной функции, если он остаётся полезен вне live journal.

Практическое решение по scope:

- `Rerun full run` трактовать как создание нового run по `project_id` без
  наследования старого subset;
- live event history в памяти UI убрать, но event recorder можно оставить для
  `export_run_log()` и отладки; сам экспорт логов лучше строить напрямую из recorder, а не через отдельный live journal в состоянии UI;
- обратную совместимость делать на уровне загрузки `Run`, а не через
  агрессивное переписывание всех enum/serializer-механизмов.

## Definition of Done

- В UI остаются только run-действия `Start`, `Cancel`, `Rerun full run`.
- В коде нет публичной логики `pause`, `resume`, `retry_failed_only`,
  `rerun_selected`, `rerun_current_paragraph`.
- В новых run используется только простой набор статусов:
  `running`, `cancelled`, `completed`, `failed`.
- `RunCheckpoint` и `Run.checkpoint` больше не нужны runtime-коду и отсутствуют в
  новой модели.
- Live state UI больше не содержит `event_journal`.
- Polling не знает про terminal status `paused`.
- Старые сохранённые `run.json` со статусом `paused` или `ready` читаются без
  падения и нормализуются к поддерживаемому состоянию.
- Тесты и линтер проходят после удаления старых веток.

## Риски и как их не пропустить

- Риск: старый workspace сломается после удаления `RunCheckpoint`.
  Контроль: обязательный regression test на legacy `run.json`.

- Риск: `Rerun full run` случайно унаследует старый subset абзацев.
  Контроль: отдельный тест, который запускает rerun после subset-run и проверяет,
  что новый run идёт по всему проекту.

- Риск: UI-кнопки удалят, а controller/service API останутся.
  Контроль: статический поиск по старым method names после завершения cleanup.

- Риск: polling сохранит paused-terminal ветку и продолжит жить по старому
  контракту.
  Контроль: отдельный unit test на `TERMINAL_RUN_STATUSES`.

- Риск: cancel в parallel execution деградирует после удаления pause-веток.
  Контроль: сохранить и адаптировать тесты на queued/inflight cancellation.

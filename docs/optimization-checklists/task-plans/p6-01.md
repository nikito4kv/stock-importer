# P6-01. ТЗ на упрощение provider-системы

Исходный пункт плана: [phase-6-simplification-checklist.md](./phase-6-simplification-checklist.md)

Предварительное условие для UI-части: [p6-00-qt-only-ui-layer-spec-2026-03-21.md](./p6-00-qt-only-ui-layer-spec-2026-03-21.md) уже выполнен, Tk-слой удален.

## Цель

Свести desktop-приложение к фиксированному набору провайдеров и убрать код, который существует только ради универсальной "платформы провайдеров".

Целевой набор `provider_id` после P6-01:

- `storyblocks_video`
- `storyblocks_image`
- `pexels`
- `pixabay`
- `openverse`

Что удаляется полностью:

- `bing`
- `generic_web_image`
- `allow_generic_web_image`
- `wikimedia`

Важно: сейчас в коде реально поддерживаются не только `Bing`, но и `Wikimedia Commons`. Если цель формулируется как "оставить только Storyblocks + Openverse/Pixabay/Pexels", то `Wikimedia` тоже должен выйти из runtime, UI, persistence и тестов. Иначе задача будет выполнена только наполовину.

## Границы задачи

Входит в P6-01:

- runtime defaults и загрузка/сохранение настроек;
- выбор провайдеров в `domain` и `ui`;
- registry и фабрики image-provider backends;
- intent/query ветка для `generic_web_image`;
- UI-элементы и preview, где есть `Bing`, `Wikimedia` или флаг `allow_generic_web_image`;
- тесты, которые проверяют старую архитектуру с `Bing`/`Wikimedia`.

Не входит в P6-01:

- полная переделка `provider_group`/`priority`-модели; это уже P6-02;
- переписывание алгоритма выбора ассетов; в P6-01 надо только убрать удаляемые источники и связанный код;
- legacy `video_fetcher.py`, если не принято отдельное решение удалять standalone free-video tooling в Phase 6.

Практическое решение по scope:

- `free_image` в `QueryBundle` оставить как общий bucket для оставшихся бесплатных image providers;
- `provider_group` не "ломать" шире необходимого; достаточно убрать ветки `generic_web_image` и записи удаляемых провайдеров;
- `video_fetcher.py` не трогать в этом пункте, чтобы не смешивать image-cleanup и отдельную тему free-video legacy.

## Definition of Done

- В приложении нет `Bing`, `Wikimedia Commons` и чекбокса/флага `allow_generic_web_image`.
- В registry остаются только `storyblocks_video`, `storyblocks_image`, `pexels`, `pixabay`, `openverse`.
- В intent payload больше не появляется `generic_web_image`.
- Сохранённые старые `settings.json` с `bing`/`wikimedia`/`allow_generic_web_image` загружаются без падения и автоматически нормализуются до поддерживаемого набора.
- Preview, validation и run execution работают только с целевым набором providers.
- Поиск и скачивание для `Storyblocks`, `Pexels`, `Pixabay`, `Openverse` продолжают работать.

- [config/settings.py](../../config/settings.py)

Что должно получиться:

Как проверить:

Практический совет:

## Риски и как их не пропустить

- Риск: старый `settings.json` снова подтянет `bing` и `wikimedia`.
  Контроль: обязательный regression test на `SettingsRepository.load()`.

- Риск: удаление `include_generic_web_image` сломает intent serialization и ручное редактирование абзаца.
  Контроль: прогон [tests/test_paragraph_intents.py](../../tests/test_paragraph_intents.py) и сценария `update_paragraph_queries(...)` в [tests/test_ui_controller.py](../../tests/test_ui_controller.py).

- Риск: UI очистят, но runtime всё ещё будет держать старые provider ids.
  Контроль: после `apply_forms_to_settings(...)` проверять итоговый `settings.providers.enabled_providers`.

- Риск: тесты на parallel free-image mode были написаны через `openverse + wikimedia` и после удаления `wikimedia` пропадёт coverage parallel branch.

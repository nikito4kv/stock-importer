# P6-03. Детальное ТЗ: упростить настройки в интерфейсе

Источник задачи: [phase-6-simplification-checklist.md](./phase-6-simplification-checklist.md)

Предварительное условие: [p6-00-qt-only-ui-layer-spec-2026-03-21.md](./p6-00-qt-only-ui-layer-spec-2026-03-21.md) уже выполнен, Tk-слой удален.

## Цель

Сделать экран запуска понятным для обычного пользователя: запуск должен быть возможен без знания `workers`, `queue size`, `retry budget`, `relevance timeout` и других инженерных ручек.

Результат задачи:

- пользователь выбирает понятный профиль запуска: `Обычный`, `Быстрый` или `Custom`;
- технические настройки больше не занимают основной сценарий запуска;
- `Custom` остается, но сводится только к ручной настройке задержек и timeout-ов браузера/скачивания;
- скрытые pipeline-настройки продолжают применяться через внутренние preset-значения, а не через поля формы.

## Проблема в текущей реализации

Сейчас технические поля живут одновременно в UI-контракте, Qt desktop-реализации и контроллере:

- [ui/contracts.py](../../ui/contracts.py)
- [ui/controller.py](../../ui/controller.py)
- [ui/qt_app.py](../../ui/qt_app.py)

Текущая `UiAdvancedSettingsViewModel` содержит ручки, которые пользователь не должен понимать:

- `paragraph_workers`
- `provider_workers`
- `provider_queue_size`
- `download_workers`
- `download_queue_size`
- `relevance_workers`
- `relevance_queue_size`
- `queue_size`
- `search_timeout_seconds`
- `relevance_timeout_seconds`
- `retry_budget`
- `top_k_to_relevance`
- `early_stop_quality_threshold`
- `fail_fast_storyblocks_errors`
- `no_match_budget_seconds`
- `full_script_context_char_budget`
- `cache_root`
- `browser_profile_path`

Дополнительно есть смешение смыслов:

- текущий `mode_id` в [domain/project_modes.py](../../domain/project_modes.py) описывает продуктовый режим провайдеров;
- новый набор `Обычный / Быстрый / Custom` должен описывать профиль выполнения run, а не заменять `project_mode`;
- сейчас пресеты сохраняют почти полный snapshot настроек через [ui/controller.py](../../ui/controller.py), [services/settings_manager.py](../../services/settings_manager.py) и [storage/repositories.py](../../storage/repositories.py), из-за чего в пресеты попадает лишняя техническая конфигурация.

## Точное продуктовое решение

### 1. Разделить два понятия

Нужно явно разделить:

- `Режим проекта` (`project_mode`) - что искать: Storyblocks video, Storyblocks images, free images и т.д.;
- `Профиль запуска` (`launch_profile`) - как агрессивно выполнять run.

Новый `launch_profile` не заменяет и не переименовывает `mode_id` из [domain/project_modes.py](../../domain/project_modes.py).

### 2. Новый профиль запуска

В форме запуска должен появиться новый пользовательский переключатель:

- `Обычный`
- `Быстрый`
- `Custom`

Рекомендуемое имя поля в UI: `Профиль запуска`.

Рекомендуемое техническое имя:

- `launch_profile_id: Literal["normal", "fast", "custom"]`

Рекомендуемое место в модели:

- добавить поле в [ui/contracts.py](../../ui/contracts.py) в `UiQuickLaunchSettingsViewModel`;
- не хранить `launch_profile_id` в `ApplicationSettings`, а вычислять его из сохраненных runtime-настроек или из preset payload.

Причина: это UI-level понятие. Сохранять его в глобальном приложенческом конфиге не нужно, если контроллер умеет:

- вывести `normal` из набора дефолтных значений;
- вывести `fast` из набора fast-значений;
- все остальные случаи считать `custom`.

### 3. Что должно остаться доступно пользователю

Оставить в пользовательском сценарии:

- `Режим проекта`
- выбор абзацев
- количество итоговых supporting/fallback изображений
- ручной prompt Gemini
- `attach_full_script_context`
- выбор бесплатных image providers
- API keys
- Storyblocks session actions
- presets
- новый `Профиль запуска`

### 4. Что должно быть убрано из обычного UI

Из формы нельзя больше напрямую редактировать:

- количество потоков;
- размеры очередей;
- search/relevance timeout-ы pipeline;
- retry budget;
- `top_k_to_relevance`;
- `early_stop_quality_threshold`;
- `fail_fast_storyblocks_errors`;
- `no_match_budget_seconds`;
- `full_script_context_char_budget`;
- `cache_root`;
- `browser_profile_path`.

### 5. Что остается в `Custom`

`Custom` должен позволять только настройку задержек и timeout-ов, которые пользователь реально может использовать для обхода медленного/нестабильного скачивания:

- `action_delay_ms`
- `launch_timeout_ms`
- `navigation_timeout_ms`
- `downloads_timeout_seconds`

Точное правило:

- если `launch_profile_id != "custom"`, эти поля не отображаются или отображаются disabled/read-only;
- если `launch_profile_id == "custom"`, эти поля становятся доступными для редактирования;
- отдельный `slow_mode` checkbox убрать из UI;
- `slow_mode` вычислять автоматически как `action_delay_ms > 0`.

## Внутренние preset-значения профилей

Ниже описан обязательный source of truth для скрытых технических настроек. Пользователь эти значения не редактирует напрямую.

### Профиль `normal`

Браузер:

- `action_delay_ms = 900`
- `slow_mode = True`
- `launch_timeout_ms = 45000`
- `navigation_timeout_ms = 30000`
- `downloads_timeout_seconds = 120.0`

Pipeline:

- `provider_workers = 4`
- `provider_queue_size = 8`
- `download_workers = 4`
- `download_queue_size = 8`
- `relevance_workers = 2`
- `relevance_queue_size = 8`
- `search_timeout_seconds = 20.0`
- `relevance_timeout_seconds = 10.0`
- `retry_budget = 2`
- `top_k_to_relevance = 24`
- `early_stop_quality_threshold = 8.0`
- `fail_fast_storyblocks_errors = True`
- `no_match_budget_seconds = 20.0`

Оркестратор:

- для `storyblocks_safe` и `mixed_safe`: `paragraph_workers = 1`, `queue_size = 1`
- для `free_images_parallel`: `paragraph_workers = 2`, `queue_size = 2`

### Профиль `fast`

Браузер:

- `action_delay_ms = 0`
- `slow_mode = False`
- `launch_timeout_ms = 30000`
- `navigation_timeout_ms = 20000`
- `downloads_timeout_seconds = 90.0`

Pipeline:

- `provider_workers = 4`
- `provider_queue_size = 8`
- `download_workers = 4`
- `download_queue_size = 8`
- `relevance_workers = 2`
- `relevance_queue_size = 8`
- `search_timeout_seconds = 12.0`
- `relevance_timeout_seconds = 6.0`
- `retry_budget = 1`
- `top_k_to_relevance = 24`
- `early_stop_quality_threshold = 8.0`
- `fail_fast_storyblocks_errors = True`
- `no_match_budget_seconds = 10.0`

Оркестратор:

- для `storyblocks_safe` и `mixed_safe`: `paragraph_workers = 1`, `queue_size = 1`
- для `free_images_parallel`: `paragraph_workers = 4`, `queue_size = 4`

### Профиль `custom`

База:

- за основу берется профиль `normal`;
- затем применяются только пользовательские override для:
  - `action_delay_ms`
  - `launch_timeout_ms`
  - `navigation_timeout_ms`
  - `downloads_timeout_seconds`

Важно:

- `Custom` не должен возвращать скрытые pipeline knobs в UI;
- `Custom` не должен позволять настраивать `paragraph_workers`, `queue_size`, `retry_budget` и похожие поля.

### Настройки, preset-ы и backward compatibility

Изменить:

- [config/settings.py](../../config/settings.py)
- [storage/repositories.py](../../storage/repositories.py)
- [services/settings_manager.py](../../services/settings_manager.py)
- [domain/models.py](../../domain/models.py)

Что сделать:

- не удалять из `ApplicationSettings` те runtime-поля, которые еще реально используются pipeline;
- удалить только UI-поля, которые не влияют на runtime вообще:
  - `cache_root` из UI advanced model;
  - `browser_profile_path` из UI advanced model;
- в preset-ах перейти на компактный user-facing snapshot, например:
  - `quick_launch`
  - `launch_profile_id`
  - `custom_timing_overrides` для `custom`
- legacy preset-ы с полным `settings_snapshot` должны продолжать загружаться.

Минимальное требование совместимости:

- старый preset не должен падать на загрузке;
- если старый preset совпадает с `normal`, форма открывается как `normal`;
- если старый preset совпадает с `fast`, форма открывается как `fast`;
- иначе форма открывается как `custom`, а пользовательские timeout-ы переносятся в 4 поля `Custom`.

Практическое правило:

- не менять `Preset` как доменную сущность, если достаточно нормализовать `settings_snapshot` на границе загрузки;
- не тащить в новый preset payload `storage`, `security`, пути workspace и другие environment-specific поля.

Что должно получиться:

- preset-ы перестают хранить инфраструктурный мусор;
- пользовательские preset-ы становятся переносимее;
- старые preset-ы не ломают запуск после обновления.

Как проверить:

- unit test: загрузка legacy preset-а с `browser.slow_mode/action_delay_ms` не падает;
- unit test: новый preset не содержит `storage`, `security`, `cache_root`, `browser_profile_path`;
- unit test: round-trip `save -> export -> import -> load` сохраняет `launch_profile_id` и `custom` timing values.

## Acceptance criteria

- основной экран запуска стал короче и понятнее;
- пользователь может выбрать `Обычный`, `Быстрый` или `Custom`, не разбираясь в потоках и очередях;
- `Custom` позволяет вручную править только delay/timeout настройки;
- Storyblocks-safe guard не сломан и по-прежнему запрещает опасный paragraph parallelism;
- legacy preset-ы продолжают открываться;
- новые preset-ы не сериализуют environment-specific и скрытые технические поля;
- Qt использует компактный view-model contract без legacy advanced knobs;
- документация не обучает пользователя настройке `workers/queues/retry budget`.

## Практические советы

- Не прячьте старые технические поля просто через `setVisible(False)`. Если поле больше не является частью пользовательского контракта, его нужно убрать из `UiAdvancedSettingsViewModel`, формы и preset schema.
- Не дублируйте profile constants между resolver-ом и `ui/qt_app.py`. Один resolver, одна таблица значений.
- Не смешивайте `project_mode` и `launch_profile`. Это два разных выбора, у них разные последствия.
- Если задача пойдет поэтапно, сначала сделайте controller/profile resolver и тесты, потом меняйте Qt. Иначе легко сломать совместимость preset-ов.
- Если в процессе выяснится, что pipeline все еще жестко зависит от удаляемых ручек, скрывайте эти параметры от пользователя, но оставляйте как internal constants до отдельной cleanup-задачи. Не возвращайте их в UI.

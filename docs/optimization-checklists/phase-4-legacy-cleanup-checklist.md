# Фаза 4: Очистка Legacy И Мертвого Кода

Цель фазы: оставить функционал, но убрать дублирование и неканоничные runtime-пути.

## P4-01 [ ] Вынести legacy CLI из корня репозитория

Действия:
1. Создать пакет `legacy_cli/`.
2. Переместить `image_fetcher.py`, `video_fetcher.py`, `keyword_extractor.py` в пакет.
3. В корне оставить thin wrappers с предупреждением о legacy статусе.

Артефакты:
- `legacy_cli/*`
- root wrappers `image_fetcher.py`, `video_fetcher.py`, `keyword_extractor.py`

Проверка:
1. CLI команды запускаются как раньше.
2. Новый код не зависит от legacy CLI модулей.

## P4-02 [ ] Формализовать adapter boundary для legacy_core

Действия:
1. Создать `legacy_adapters/` для всех разрешенных мостов в legacy.
2. Заменить прямые импорты `legacy_core` в new-core на импорты адаптеров.
3. Запретить новые прямые импорты правилом в тестах/линтере.

Артефакты:
- `legacy_adapters/*`
- `pipeline/*`, `providers/*` (переход на адаптеры)

Проверка:
1. В new-core нет прямых `from legacy_core ...` (кроме адаптеров).
2. Поведение существующих сценариев не меняется.

## P4-03 [ ] Мигрировать ingestion из legacy_core в new-core

Действия:
1. Перенести DOCX parsing/validation в `pipeline/ingestion.py` или `services/ingestion`.
2. Сохранить текущие бизнес-правила нумерации и ошибок.
3. Закрыть миграцию regression-тестами.

Артефакты:
- `pipeline/ingestion.py`
- `tests/test_ingestion.py`

Проверка:
1. Результаты ingest идентичны baseline на существующих фикстурах.
2. Удалена runtime-зависимость от `legacy_core.ingestion`.

## P4-04 [ ] Мигрировать общие утилиты query/license/common

Действия:
1. Перенести используемые утилиты в new-core (`providers`/`services`).
2. Сохранить API-совместимость через адаптерный слой.
3. Удалить дубли и неиспользуемые legacy функции.

Артефакты:
- `providers/images/filtering.py`
- `providers/images/querying.py`
- `pipeline/intents.py`

Проверка:
1. Поведение ранжирования и фильтрации совпадает с baseline.
2. Список активных legacy импортов сокращен.

## P4-05 [ ] Удалить неиспользуемые настройки и поля

Действия:
1. Провести аудит полей конфигурации: читается/влияет на поведение.
2. Для неиспользуемых полей: deprecate -> migration -> remove.
3. Обновить сериализацию и обратную совместимость.

Артефакты:
- `config/settings.py`
- `storage/repositories.py`
- `ui/controller.py` (если поле отображалось)

Проверка:
1. Нет "мертвых" полей в конфиге, которые только сериализуются.
2. Старые `settings.json` продолжают корректно загружаться.

## P4-06 [ ] Сократить дублирующий UI слой (по бизнес-решению)

Действия:
1. Принять решение: поддерживать и Qt, и Tk, или оставить одну платформу.
2. Если выбран один стек, перевести второй в maintenance/deprecated режим.
3. Удалить дублирующую логику после переходного периода.

Артефакты:
- `ui/qt_app.py`
- `ui/tk_app.py`
- `README.md`

Проверка:
1. Поддерживаемая UI ветка полностью покрывает нужные флоу.
2. Объем дублируемого UI-кода снижен.

## P4-07 [ ] Очистить неактуальные документы и ссылки

Действия:
1. Найти ссылки на отсутствующие директории/файлы (`docs/phase-*`, `docs/ai` и т.д.).
2. Обновить README/CONTRIBUTING/тесты релиза под фактическую структуру.
3. Добавить smoke-проверку валидности ссылок в docs.

Артефакты:
- `README.md`
- `CONTRIBUTING.md`
- `tests/test_phase10_release.py`

Проверка:
1. Нет ссылок на отсутствующие документы.
2. Тесты release не падают из-за документного drift.

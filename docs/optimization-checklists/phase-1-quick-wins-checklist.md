# Фаза 1: Быстрые Победы

Цель фазы: резко снизить лишний I/O и CPU overhead без изменения бизнес-функций.

## P1-01 [x] Ввести guard конкурентности для Storyblocks режима

Действия:
1. На старте run определить, используется ли Storyblocks (`storyblocks_video`/`storyblocks_image`).
2. Если да и `paragraph_workers > 1`, блокировать запуск с понятной ошибкой.
3. Добавить сообщение в UI с рекомендацией автокоррекции до `1`.

Артефакты:
- `ui/controller.py`
- `pipeline/media.py` или `app/runtime.py` (точка валидации)

Проверка:
1. Storyblocks run не стартует с небезопасным параллелизмом.
2. Free-image режимы продолжают работать с `paragraph_workers > 1`.

## P1-02 [x] Уменьшить стоимость polling в UI

Действия:
1. Разделить polling на lightweight (live status) и heavy refresh (полное состояние).
2. Heavy refresh выполнять только при terminal state или по явному действию.
3. Сделать интервал polling адаптивным: активный run/idle.

Артефакты:
- `ui/qt_app.py`
- `ui/tk_app.py`
- `ui/controller.py`

Проверка:
1. Во время активного run UI остается отзывчивым.
2. Количество полных `build_state` вызовов заметно снижено.

## P1-03 [x] Добавить компактный live-snapshot API

Действия:
1. В контроллере выделить метод, возвращающий только данные live-экрана.
2. Исключить из live-снимка тяжелые поля (полные списки/историю, если не нужны).
3. Заменить polling-код UI на новый lightweight метод.

Артефакты:
- `ui/controller.py`
- `ui/contracts.py`

Проверка:
1. На polling больше не выполняются дорогостоящие чтения всех run/project списков.
2. Функционально live экран не деградировал.

## P1-04 [x] Перейти на batched persist для manifest

Действия:
1. Убрать избыточные промежуточные `save_manifest` внутри одной обработки абзаца.
2. Сохранять manifest один раз по завершению абзаца и на ошибке.
3. Добавить flush в критических точках pause/cancel/finalize.

Артефакты:
- `pipeline/media.py`

Проверка:
1. Количество операций записи manifest на абзац уменьшено минимум в 2 раза.
2. После crash состояние остается консистентным.

## P1-05 [x] Перейти на batched persist для run/checkpoint

Действия:
1. В orchestrator уменьшить частоту `run_repository.save`.
2. Сохранять run/checkpoint по завершению абзаца и на state transition.
3. Убедиться, что pause/resume/retry работают без потери прогресса.

Артефакты:
- `pipeline/orchestrator.py`

Проверка:
1. Снижен I/O на длительных run.
2. Поведение pause/resume/retry не изменилось для пользователя.

## P1-06 [x] Буферизовать event logging

Действия:
1. Заменить "open file per event" на буфер + периодический flush.
2. Обеспечить flush при завершении run и при аварийном завершении.
3. Добавить ограничение памяти буфера.

Артефакты:
- `services/logging.py`
- `app/bootstrap.py`

Проверка:
1. Лог не теряется при штатном завершении.
2. Среднее время обработки event уменьшается.

## P1-07 [x] Ввести индекс manifest entry по paragraph_no

Действия:
1. Создать структуру `dict[int, ParagraphManifestEntry]` при загрузке/создании manifest.
2. Заменить линейные поиски на O(1) доступ.
3. Синхронизировать индекс при изменении manifest.

Артефакты:
- `pipeline/media.py`

Проверка:
1. Нет линейного сканирования entry в hot-path.
2. Все тесты на выбор/lock paragraph проходят.

## P1-08 [x] Сделать deduper инкрементальным

Действия:
1. Инициализировать deduper один раз на run или на manifest snapshot.
2. Обновлять deduper только новыми выбранными assets.
3. Убрать полный rebuild deduper на каждый абзац.

Артефакты:
- `pipeline/media.py`

Проверка:
1. Время обработки поздних абзацев не деградирует из-за O(n²) поведения.
2. Качество дедупликации не ухудшилось.
